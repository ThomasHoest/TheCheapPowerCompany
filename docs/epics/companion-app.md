## Overview

This component delivers the Companion iOS App for The Cheap Power Company: a SwiftUI iPhone application (iOS 15+, Danish) that lets customers sign in with MitID via Signicat, view their current spot price, running bill, and next MobilePay charge, drill into hourly price detail and consumption, manage their subscription, and receive push notifications for payment events and price alerts. The work is split across nine epics: Project Setup & Architecture (E-01) is the prerequisite for all subsequent epics; Authentication (E-02) gates Home, Price Detail, Consumption, Account, and Push Notifications; Accessibility & Polish (E-08) and Error & Offline Handling (E-09) run in parallel with the feature epics. ADR-001 section 6.1 (severity HIGH) requires that returning-user MitID logins request `scope=openid profile` (without `nin`), while new-user signup continues to request `scope=openid profile nin`; this correction is reflected in E-02 task acceptance criteria, and implementers must follow the ADR rather than unrevised spec text.

## E-02 — Authentication — MitID Login

Customers can sign in with MitID via Signicat, the app exchanges the authorization code for tokens, posts the ID token to the TCPC backend, stores the resulting TCPC session JWT and refresh token in Keychain, and renews the session silently on subsequent launches. Depends on E-01 (all tasks).

- [ ] **T-0201** Build the Login screen as the first screen on a fresh install, showing the TCPC logo, a stylised faded line drawing of Denmark (rendered at <30% opacity, hidden from VoiceOver focus order), one line of marketing copy from `Localizable.strings` (Danish), and a single "Log ind med MitID" button using brand green from T-0106 with 44x44pt minimum tappable size; nothing else interactive; renders correctly on iOS 15 and iOS 17 simulators in both light and dark mode; fully Dynamic Type compatible up to `accessibility3`; tapping the button currently triggers a stub function (real flow lands in T-0203); SwiftUI preview snapshots committed and a VoiceOver walk-through recorded in the PR. Spec coverage: US-01 AC 1; Section 8 (Imagery, Tone of voice).
- [ ] **T-0202** Configure AppAuth-iOS to fetch the Signicat discovery document at `https://<tenant>.signicat.com/auth/open/.well-known/openid-configuration` and prepare an `OIDServiceConfiguration`; load Signicat tenant URL and client ID from a build configuration file (different values for staging and production); perform discovery lazily on first auth attempt and cache for the lifetime of the app process; generate per-attempt PKCE `code_verifier` of at least 43 secure random bytes with `code_challenge_method=S256`; build and URL-encode an `idp_params` query parameter containing exactly `{"mitid":{"enable_app_switch":true,"app_switch_os":"ios","app_switch_url":"https://app.cheappower.dk/oauth/callback"}}`; per ADR-001 §6.1 select scope based on a Keychain "has-completed-signup" flag set after the first successful `POST /v1/auth/session` whose response indicates onboarding complete — first-time signup flows request `scope=openid profile nin`, returning-user logins request `scope=openid profile` (without `nin`); unit tests verify the constructed authorization URL contains all required parameters including the correct scope for each flow; PR description references ADR-001 §6.1 explicitly. Spec coverage: US-01 AC 2; Section 6 steps 3, 4 (corrected per ADR-001 §6.1); Section 12.
- [ ] **T-0203** Wire the "Log ind med MitID" button to start an `ASWebAuthenticationSession` with the URL built in T-0202, setting `prefersEphemeralWebBrowserSession = false` to allow Signicat session cookies to persist for silent re-auth; if the MitID app is installed, the system app-switches to MitID (manually verified on a device with the MitID test app), otherwise the Signicat-hosted browser flow is shown inside the auth session; on successful return, `ASWebAuthenticationSession` calls back with the redirect URL containing `?code=...&state=...`; user cancellation (closing the auth session) returns control to the Login screen with no error displayed; `WKWebView` is not present anywhere in the auth path (verified by grep in CI); manual end-to-end test on a device using the Signicat sandbox and MitID test app recorded in the PR; cancellation path verified manually. - [ ] **T-0204** Wire the Universal Link callback into the auth flow so that when iOS delivers the callback URL (either through `ASWebAuthenticationSession`'s completion handler or via `onOpenURL` if the session has already closed), the URL is passed to AppAuth-iOS for code-to-token exchange via `OIDAuthState.authState(byPresenting:...)` (or equivalent), exchanging the authorization code for ID token, access token, and refresh token at Signicat's token endpoint; on exchange success, the ID token is passed to T-0205; on exchange failure, the user is returned to the Login screen with the verbatim inline error "Login mislykkedes. Prøv igen, eller kontakt support." with a mailto link; the correlation ID from the failure is logged to the backend via `POST /v1/diagnostics` (or agreed diagnostics route) and shown in small grey type underneath the error message; end-to-end manual test with the Signicat sandbox; failure path verified by injecting an invalid `code` value. Spec coverage: US-01 AC 4; Section 6 steps 9–10; Error States row "MitID auth genuinely fails".
- [ ] **T-0205** Once AppAuth-iOS has the ID token, post it to `POST /v1/auth/session` via the network layer (T-0105) with body `{"idToken":"<jwt>"}`; on 2xx, write both tokens (TCPC session JWT and refresh token) from the response to Keychain via T-0104 and set the "has-completed-signup" Keychain flag if `onboardingComplete == true`; route the user to Home (E-03) when onboarding is complete, or to a placeholder Onboarding screen ("Onboarding kommer her") when not — full implementation belongs to a separate Onboarding epic; total time from "Log ind med MitID" tap to Home for a returning user with the MitID app installed is under 15 seconds on a standard 4G connection (manually measured on at least three attempts and recorded in the PR); verified end-to-end against the Signicat sandbox and a staging backend. Spec coverage: US-01 AC 5–7; Section 6 steps 11–12.
- [ ] **T-0206** On every app launch, if a TCPC refresh token is in Keychain, attempt a silent refresh via `POST /v1/auth/refresh` (or agreed route — confirm with backend) before the first Home-screen network call; on 2xx, replace the session JWT in Keychain and proceed to Home with no visible interruption; on 4xx (token expired/invalid), clear Keychain and show the Login screen with no error modal; on network failure, show the cached Home with the offline pill (E-09 owns the pill UI; this task surfaces the offline flag); sessions remain valid for a minimum of 30 days of inactivity (verified by simulating a 30-day-old refresh token in staging via clock manipulation or token fixtures). ## Open Questions

These questions are inherited from the companion-app spec Section 11 and from ADR-001. They affect implementation plans across multiple epics and should be resolved before the relevant epic begins.

1. **Onboarding flow detail (spec OQ #11).** T-0205 routes new users to a placeholder Onboarding screen. The full Onboarding sequence (sign-agreement, choose-interval, MobilePay-agreement creation, Eloverblik-consent) is owned by a separate Onboarding spec/epic that does not yet exist. Owner: Product. Default assumption: a separate Onboarding spec follows; this iOS plan covers only the placeholder.
2. **`suggestMaxAmount` value (ADR-001 §6.4, spec OQ #9).** Affects what is shown in Account T-0603. Owner: Product + Finance. Default per ADR-001: 3,000 DKK (from customer-onboarding.md US-05) until formally resolved. Reflect the resolved value when the backend sends it via `GET /v1/subscription`.
3. **Eloverblik consent UX (spec OQ #7).** Affects T-0505. Owner: Backend lead. Default assumption: separate Eloverblik flow, surfaced in `ASWebAuthenticationSession`.
4. **Price-area determination during onboarding (spec OQ #8).** During the brief onboarding window before metering data is available, the app should hide the Home price tile and show "Vi henter dine data". Affects T-0302 acceptance criteria. Owner: Backend lead.
5. **Push notification copy (spec OQ #10).** Drafts in section 5.6 are used verbatim in T-0703, T-0704, T-0705. Owner: Marketing/Copy. Default: ship with the drafts.
6. **Logout flag preservation (T-0207).** Whether the "has-completed-signup" Keychain flag persists across logout, or is cleared on logout. Determines whether the next sign-in requests `nin` again. Owner: Backend lead. Default: preserve across logout so the next sign-in remains a returning-user flow.
7. **Watch / widget / Live Activities scope (spec OQ #1, #2, #5).** Excluded from v1. Will be revisited for v1.1+.
