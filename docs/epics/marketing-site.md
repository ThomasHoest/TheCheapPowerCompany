# Marketing Site — Implementation Epics

**Source spec:** `docs/specifications/marketing-site.md`
**Architecture reference:** `docs/specifications/ADR-001-system-architecture.md` (section 3.4)
**Target platform:** Public marketing website (responsive, SSR/static web)
**Primary language:** Danish (da-DK)

This document breaks the Marketing Site specification into eight implementation
epics. Each epic lists its goal, dependencies, and a set of tasks sized for
1–3 engineering days. Each task carries acceptance criteria, a definition of
done, and a back-reference to the spec user story (or non-functional
requirement) it satisfies.

Tasks are intended to be picked up sequentially within an epic unless
otherwise noted. Cross-epic dependencies are called out per-epic.

---

## Epic 1 — Project Setup

**Goal:** Stand up a deployable, Danish-localised, design-tokenised web
project with the chosen framework and host, configured to meet the LCP < 2.5s
and WCAG 2.1 AA baselines from day one.

**Dependencies:**
- Open question 6 (hosting choice) defaulted to Vercel or Netlify.
- Backend price API contract from ADR-001 §3.4 (consumed in Epic 6, but the
  HTTP base URL must be known at project setup so environment variables can
  be wired in).

**Tasks:**

### T1.1 — Choose framework and bootstrap the repo

- **Description:** Decide between Next.js (App Router, RSC) and Astro for
  the marketing site. The framework must support SSR or static HTML for all
  marketing content (NFR: server-rendered HTML for SEO) and lightweight
  client islands for the live-rate widget.
- **Acceptance criteria:**
  - [ ] One framework is selected and rationale recorded in
        `docs/decisions/` or as an ADR addendum.
  - [ ] A new repo (or `apps/marketing-site` workspace) exists with the
        framework's starter template.
  - [ ] `pnpm dev` (or chosen package manager equivalent) serves the
        starter at `localhost:3000`.
  - [ ] All marketing content routes are server-rendered or statically
        generated by default (no client-only pages).
  - [ ] Node version is pinned via `.nvmrc` or `engines` in `package.json`.
- **Definition of done:** PR merged with framework choice committed,
  rationale documented, working local dev server, CI lint/build passes.
- **Spec reference:** NFR — "Server-rendered HTML for all marketing
  content."

### T1.2 — Configure hosting and deploy preview pipeline

- **Description:** Connect the repo to Vercel (or Netlify) so every PR
  gets a preview URL and `main` deploys to a staging environment. Set up
  a custom-domain placeholder (e.g. `staging.tcpc.dk`).
- **Acceptance criteria:**
  - [ ] PRs automatically get a preview URL posted as a comment.
  - [ ] `main` branch auto-deploys to a staging URL.
  - [ ] Environment variables can be configured per-environment in the
        host's UI.
  - [ ] HTTPS is enforced; HTTP redirects to HTTPS.
  - [ ] A health-check route (`/health` or equivalent) returns 200.
- **Definition of done:** Two consecutive PRs have produced working
  preview URLs; staging URL serves the starter page over HTTPS.
- **Spec reference:** Technical Context — "Hosting: Static-site /
  SSR-capable host."

### T1.3 — Danish locale configuration

- **Description:** Configure the site for `da-DK` as the sole locale.
  HTML `lang="da"`, date/number/currency formatting via `Intl` with
  `da-DK`, and routing that does not require a locale segment in the URL.
- **Acceptance criteria:**
  - [ ] `<html lang="da">` is emitted on every rendered page.
  - [ ] A shared formatting utility exposes
        `formatCurrency`, `formatNumber`, and `formatTime` bound to `da-DK`.
  - [ ] An example page renders "1.234,56 kr." (Danish thousand and
        decimal separators) using the utility.
  - [ ] No URL contains `/da/` or similar locale prefix.
  - [ ] All copy strings live in a single content module (e.g.
        `content/da.ts`) rather than being inlined per component, to make
        future v2 multi-language work tractable.
- **Definition of done:** Lint rule or test asserts every rendered page
  has `lang="da"`; formatting utility unit-tested with at least three
  inputs each.
- **Spec reference:** NFR — "Localisation: Danish (da-DK) only in v1."

### T1.4 — Design token integration

- **Description:** Implement the design token layer for the green-and-light
  modern aesthetic described in the spec's Design Guidance section. Tokens
  must cover colour, typography, spacing, radius, and shadow.
- **Acceptance criteria:**
  - [ ] Tokens are defined as CSS custom properties (or equivalent in the
        chosen styling system) and consumed via a single import.
  - [ ] Palette includes: light backgrounds (white / off-white), dark
        body text, exactly one accent green for CTAs and emphasis.
  - [ ] Typography tokens reference a modern sans-serif (Inter, Geist, or
        equivalent) loaded via `font-display: swap`.
  - [ ] Radius tokens cover 8px and 16px; shadow tokens are subtle.
  - [ ] A Storybook page or a `/styleguide` route renders the token
        palette and typography scale for visual review.
  - [ ] Contrast for body text on background and CTA text on accent green
        meets WCAG AA (>= 4.5:1 for normal text, >= 3:1 for large text).
- **Definition of done:** Design lead signs off on the styleguide page;
  automated contrast check (e.g. via a unit test using
  `wcag-contrast`) passes for the documented foreground/background pairs.
- **Spec reference:** Design Guidance — "primarily light… green as the
  accent and CTA colour."

### T1.5 — Performance budget and LCP guardrail

- **Description:** Wire performance budgets into CI so a regression below
  LCP 2.5s or above 500 KB total transferred page weight fails the build.
- **Acceptance criteria:**
  - [ ] Lighthouse CI (or equivalent) runs against every PR preview.
  - [ ] Build fails if mobile LCP > 2.5s, mobile Performance score < 90,
        or Accessibility score < 95.
  - [ ] Build fails if total transferred page weight on `/` exceeds 500
        KB (excluding the optional sparkline).
  - [ ] Failures are reported as a PR comment with the regressed metric.
- **Definition of done:** A deliberate regression PR (e.g. importing a
  large image) is shown to fail the budget; the budget passes on the
  starter.
- **Spec reference:** NFR — "Performance: LCP under 2.5s; total
  transferred page weight under 500 KB"; US-05 — "Lighthouse Mobile
  Performance score of at least 90 and Accessibility score of at least 95."

### T1.6 — Accessibility tooling baseline

- **Description:** Configure axe-core (or equivalent) automated a11y
  testing in CI and developer tooling so violations are caught early.
- **Acceptance criteria:**
  - [ ] `axe-core` runs against every page in CI as part of an
        end-to-end test suite (Playwright or equivalent).
  - [ ] Build fails on any `serious` or `critical` axe violation.
  - [ ] An ESLint plugin (e.g. `eslint-plugin-jsx-a11y` for React, or the
        Astro equivalent) is enabled with the recommended ruleset.
  - [ ] Documentation in `README.md` describes how to run a11y checks
        locally.
- **Definition of done:** Sample component with a known violation
  (e.g. missing `alt`) is shown to fail CI; the starter passes clean.
- **Spec reference:** NFR — "WCAG 2.1 AA. Lighthouse a11y score >= 95."

## Epic 2 — Landing Page

**Goal:** Build the homepage at `/` so a Danish visitor on a 390x844 mobile
device sees brand, value proposition, live rate, and primary CTA above the
fold, and below-the-fold sees pricing explanation and how-it-works.

**Dependencies:**
- Epic 1 complete (framework, locale, tokens, perf guardrails).
- Epic 6 T6.1 must be available for the live rate to render in the hero —
  if Epic 6 is not yet built, the hero ships with the placeholder text
  "Aktuel pris hentes…" wired to the same component contract.

**Tasks:**

### T2.1 — Hero section with value proposition and CTA

- **Description:** Build the hero block: brand mark, single-line Danish
  value proposition, primary "Tilmeld dig" CTA, slot for the live-rate
  widget. Mobile-first layout reflowing to two-column on desktop.
- **Acceptance criteria:**
  - [ ] On a 390x844 viewport, the brand name, value proposition, live
        rate slot, and CTA all fit above the fold without scrolling.
  - [ ] On a 1440x900 viewport, the same elements are visible without
        scrolling.
  - [ ] The CTA is a single styled button with the label "Tilmeld dig"
        and routes to the onboarding entry URL (Epic 4).
  - [ ] CTA is keyboard-focusable; tab order places it after the brand
        link and before below-the-fold sections.
  - [ ] Headline uses the largest typography token; sub-headline uses
        the second-largest.
  - [ ] No carousel, no autoplay video, no parallax.
- **Definition of done:** Visual regression snapshot stored for mobile
  and desktop; product owner approves the copy in Danish; viewport
  fold assertions are covered by an end-to-end test.
- **Spec reference:** US-01 — "Visitor lands on the homepage and
  understands the offering."

### T2.2 — Pricing explanation section

- **Description:** Build the section that states the per-kWh markup
  (concrete øre/kWh number), the monthly subscription fee in DKK, the
  weekly fee in DKK, the recommended default, the binding-period
  statement, and a worked annual example.
- **Acceptance criteria:**
  - [ ] Section is reachable from the landing page in one click via an
        in-page anchor link or top-nav entry labelled "Priser".
  - [ ] Per-kWh markup is rendered from a single content constant (so
        the number can be changed in one place when Open Question 1 is
        resolved).
  - [ ] Monthly and weekly subscription fees are rendered from content
        constants. The recommended option is visually distinguished.
  - [ ] A worked example is shown using the spec's template
        ("Et husstand der bruger 4.000 kWh/år betaler ca. …"). The
        computed example value is derived from the same constants, not
        hard-coded.
  - [ ] A prominent statement of binding terms appears (default copy:
        "Ingen binding. Ingen skjulte gebyrer. Ingen udmeldelsesgebyr.").
  - [ ] A repeated "Tilmeld dig" CTA appears at the end of the section.
- **Definition of done:** Constants documented in
  `content/pricing.ts` (or equivalent) with a comment naming each open
  question that gates the final values; section renders correctly at
  360px, 768px, and 1440px widths.
- **Spec reference:** US-02 — "Visitor reads about pricing."

### T2.3 — How-it-works section (three steps)

- **Description:** Build the three-step explainer: (1) Tilmeld med MitID,
  (2) Godkend MobilePay-abonnement, (3) Få strøm til spotpris + lille
  tillæg.
- **Acceptance criteria:**
  - [ ] Three numbered steps render in order with Danish copy matching
        the spec.
  - [ ] Each step has a heading and a short paragraph (max 2 sentences).
  - [ ] On mobile the steps stack vertically; on desktop they may
        render as a three-column grid.
  - [ ] No external imagery is required; iconography uses inline SVG
        from the design token library.
- **Definition of done:** Section renders as specified; copy reviewed
  by product owner; visual regression snapshot stored.
- **Spec reference:** Pages and Content table — "How it works section."

### T2.4 — Sticky mobile CTA

- **Description:** Implement a sticky bottom CTA bar visible on mobile
  long-scroll pages so the "Tilmeld dig" button never leaves the
  viewport once the user scrolls past the hero.
- **Acceptance criteria:**
  - [ ] On viewports < 768px wide, after scrolling past the hero, a
        sticky bar appears at the bottom containing only the
        "Tilmeld dig" button.
  - [ ] The sticky bar has a tap target of at least 44x44 px.
  - [ ] On viewports >= 768px wide, no sticky bar is shown.
  - [ ] The sticky bar respects iOS safe-area insets.
  - [ ] The sticky bar is hidden when the in-page CTA is in the
        viewport (to avoid double-CTA noise).
- **Definition of done:** Manual test on a real iPhone confirms safe
  area handling; end-to-end test asserts visibility behaviour at three
  scroll positions on a mobile viewport.
- **Spec reference:** US-05 — "The signup CTA remains visible or sticky
  on long-scroll mobile views."

### T2.5 — Top-nav with single CTA

- **Description:** Build the top navigation: brand mark linking to `/`,
  in-page anchors to "Priser" and "Sådan virker det", and the single
  primary CTA "Tilmeld dig". Same nav appears on every page.
- **Acceptance criteria:**
  - [ ] Nav is rendered server-side as part of the layout, not a
        client-only component.
  - [ ] On mobile, a hamburger menu (or simple stacked layout) is used;
        the CTA remains visible without opening the menu.
  - [ ] All anchor links use `#priser`, `#sadan-virker-det`, etc., and
        scroll smoothly to the corresponding sections.
  - [ ] Tab order: brand, anchors, CTA.
  - [ ] No second competing CTA colour appears anywhere in the nav.
- **Definition of done:** Nav is consumed by every page in Epics 2 and
  5; keyboard navigation tested; visual regression snapshot stored.
- **Spec reference:** US-04 — "Primary CTA button is present in the top
  navigation"; Resolved Decisions — "One label, one colour, repeated."

## Epic 3 — Companion App Teaser

**Goal:** Reassure the visitor that there is a companion app, describe
its two key features, and place App Store / Play Store badges (or a
"Coming soon" state if not yet published).

**Dependencies:**
- Epic 1 complete.
- Open question 4 (App Store availability at launch) — default
  assumption: "Coming soon" badges.

**Tasks:**

### T3.1 — App teaser block

- **Description:** Build the companion app teaser section on the
  landing page. Includes a screenshot mockup placeholder, a short
  Danish headline, a two-bullet feature list (live kWh rate, current
  bill), and the App Store badge slot.
- **Acceptance criteria:**
  - [ ] Section appears on the landing page below the how-it-works
        section and above the FAQ.
  - [ ] Screenshot slot accepts a single image asset; layout does not
        break if the image is missing or fails to load (alt text
        fallback rendered).
  - [ ] Two feature bullets are rendered in Danish:
        "Se din aktuelle kWh-pris i realtid" and
        "Se din løbende regning"  (final copy to be confirmed by product).
  - [ ] Section is readable on 360px wide screens with no horizontal
        scroll.
- **Definition of done:** Section renders with a placeholder
  screenshot; product owner approves copy; a11y check passes on the
  rendered section.
- **Spec reference:** US-06 — "Visitor learns about the companion app."

### T3.2 — App Store badge component with "Coming soon" fallback

- **Description:** Build a reusable badge component that renders either
  the official App Store / Google Play badges (when the store URLs are
  configured) or a "Kommer snart" badge in the same visual footprint
  when they are not.
- **Acceptance criteria:**
  - [ ] Component accepts `appStoreUrl` and `playStoreUrl` props (or
        equivalent config values).
  - [ ] When a URL is provided, the badge renders as a clickable link
        opening in a new tab with `rel="noopener noreferrer"`.
  - [ ] When no URL is provided, the badge renders as a non-clickable
        "Kommer snart" pill with the same dimensions.
  - [ ] Badge images are loaded as optimised SVG or properly compressed
        PNG (per Apple/Google brand guidelines if real badges are
        used).
  - [ ] Alt text in Danish: "Hent på App Store" / "Hent på Google Play"
        / "Appen kommer snart".
- **Definition of done:** Component is used in the teaser section;
  toggling the URL config swaps between live and "Coming soon" states
  without layout shift; snapshot test covers both states.
- **Spec reference:** US-06 — "If app store links exist, they are
  present; otherwise a 'Coming soon' state is shown."

## Epic 4 — Signup CTA Flow

**Goal:** Ensure every "Tilmeld dig" CTA on the marketing site is a
plain link out to the Onboarding Flow entry URL — never an inline form
on the marketing site itself.

**Dependencies:**
- Epic 1 complete.
- Open question 7 (subdomain vs `/tilmeld` path) — default assumption:
  same domain, `/signup` path. The implementation must read the URL
  from a single environment variable so the choice can be flipped without
  code changes.

**Tasks:**

### T4.1 — Onboarding URL configuration

- **Description:** Introduce a single environment variable
  (`NEXT_PUBLIC_ONBOARDING_URL` or framework equivalent) holding the
  full URL to the onboarding entry point. All CTAs across the site
  read from this variable.
- **Acceptance criteria:**
  - [ ] All CTA buttons resolve their `href` from the env variable at
        build time (or render time for SSR).
  - [ ] Env variable is documented in `README.md` with example values
        for local, staging, production.
  - [ ] If the env variable is unset at build time, the build fails
        with a clear error message.
  - [ ] No inline form fields appear anywhere on the marketing site —
        the CTA is the only signup affordance.
- **Definition of done:** Grep across the codebase confirms no
  hard-coded `/signup` URLs; an end-to-end test asserts the CTA href
  matches the configured value.
- **Spec reference:** US-04 — "The CTA carries no inline form fields";
  Resolved Decisions — "Marketing Site has zero forms apart from the
  CTA link."

### T4.2 — CTA component with telemetry hook

- **Description:** Build a single reusable `<SignupCTA />` component
  used in every CTA placement (top nav, hero, post-pricing, post-FAQ,
  sticky mobile bar). Component fires an analytics event on click
  before navigation.
- **Acceptance criteria:**
  - [ ] Component renders an `<a>` element (not a `<button>`) so it
        works without JavaScript and is treated as a navigation by
        screen readers.
  - [ ] Component accepts a `placement` prop (e.g. `"hero"`,
        `"top-nav"`, `"sticky-mobile"`) which is included in the
        analytics event payload.
  - [ ] Click fires a `cta_clicked` analytics event with `placement`
        and the destination URL before navigation occurs.
  - [ ] Component has a single visual variant — same colour, same
        label "Tilmeld dig", same shape — across every placement.
  - [ ] Tap target is at least 44x44 px on mobile.
- **Definition of done:** Every CTA in the codebase uses this
  component; an end-to-end test clicks each placement and asserts the
  analytics event payload is recorded.
- **Spec reference:** US-04 — "Primary CTA button is present in the
  top navigation, in the hero section, and at the end of the pricing
  section."

### T4.3 — Signup-unreachable fallback page

- **Description:** Build a fallback page (e.g. `/tilmeld-utilgaengelig`)
  that the visitor lands on if the onboarding flow returns an error.
  This page is reached either by the onboarding flow redirecting back
  with an error, or by client-side handling of a connectivity failure.
- **Acceptance criteria:**
  - [ ] Page renders with the verbatim Danish copy:
        "Tilmelding er midlertidigt nede. Prøv igen om et øjeblik,
        eller skriv til support@…" (final email TBD per Open Question
        11/contact).
  - [ ] Page contains a "Prøv igen" link back to the onboarding URL
        and a "Til forsiden" link back to `/`.
  - [ ] Page is server-rendered; no JavaScript required.
  - [ ] Page returns HTTP 200 (it is an informational page, not an
        error page).
- **Definition of done:** Manual test confirms the page renders;
  end-to-end test asserts both links resolve to expected destinations.
- **Spec reference:** Error States — "Signup service / Onboarding Flow
  URL unreachable."

## Epic 5 — Supporting Pages

**Goal:** Build the FAQ, pricing/terms, privacy policy, cookie policy,
cookie consent banner, and footer so the site is legally and
informationally complete.

**Dependencies:**
- Epic 1 complete.
- Open questions 11 (CVR), 12 (FAQ content). Defaults: placeholder CVR
  and a starter FAQ drafted by the spec writer pending product review.

**Tasks:**

### T5.1 — Footer with CVR and legal links

- **Description:** Build the site-wide footer rendered on every page.
  Includes company name, CVR number, contact email, and links to
  Handelsbetingelser, Privatlivspolitik, and Cookies.
- **Acceptance criteria:**
  - [ ] Footer renders on `/`, `/handelsbetingelser`,
        `/privatlivspolitik`, `/cookies`, the FAQ page, and the 404 page.
  - [ ] CVR number is rendered from a single content constant
        (currently a placeholder) with a code comment naming Open
        Question 11.
  - [ ] All three legal links are reachable in one click from any page.
  - [ ] Footer is a `<footer>` landmark element with a labelled
        navigation list inside.
  - [ ] Footer copy is Danish.
- **Definition of done:** Lint check or end-to-end test asserts the
  footer appears on every public route; a11y check passes.
- **Spec reference:** US-07 — "Footer links to 'Handelsbetingelser' and
  'Privatlivspolitik' are present on every page."

### T5.2 — FAQ page

- **Description:** Build the FAQ section / page with 5–8 question /
  answer pairs covering: switching providers, MitID, MobilePay, what
  energinet.dk is, what happens if I move, how to cancel.
- **Acceptance criteria:**
  - [ ] FAQ page exists at `/faq` (or as an in-page section on the
        landing page — final placement decided in this task and
        documented in the PR).
  - [ ] At least 5 Q&A pairs are present, each in Danish.
  - [ ] Q&A pairs use a `<details>`/`<summary>` accordion or a
        semantically equivalent disclosure pattern with proper ARIA.
  - [ ] All accordions are keyboard-operable (Enter/Space to toggle).
  - [ ] FAQ content is sourced from a single content module.
- **Definition of done:** Product owner reviews the starter FAQ copy;
  a11y check passes; visual regression snapshot stored.
- **Spec reference:** Pages and Content table — "FAQ section: 5–8 Q&A."

### T5.3 — Handelsbetingelser (terms) page

- **Description:** Build the public terms-of-service page in Danish at
  `/handelsbetingelser`.
- **Acceptance criteria:**
  - [ ] Page renders Danish terms-of-service copy supplied by Legal
        (placeholder Lorem-ipsum-equivalent until copy lands).
  - [ ] Page is publicly accessible without authentication.
  - [ ] Page is server-rendered, no JavaScript required to read it.
  - [ ] Page uses a clear typographic hierarchy: H1, section headings,
        paragraphs.
  - [ ] Page is reachable from the footer on every page.
- **Definition of done:** Page renders with placeholder copy and is
  flagged in the PR description as "awaiting Legal copy"; a11y check
  passes.
- **Spec reference:** US-07 — "Both pages are publicly accessible
  without authentication."

### T5.4 — Privatlivspolitik (privacy) page

- **Description:** Build the public GDPR-compliant privacy policy page
  in Danish at `/privatlivspolitik`.
- **Acceptance criteria:**
  - [ ] Page renders Danish privacy-policy copy (placeholder until
        Legal copy lands).
  - [ ] Page lists the categories of data processed: analytics
        (chosen vendor), no cookies for marketing.
  - [ ] Page links to the cookie policy.
  - [ ] Page is publicly accessible, server-rendered, no JS required.
  - [ ] Page is reachable from the footer on every page.
- **Definition of done:** Page renders with placeholder copy flagged
  in the PR description as "awaiting Legal copy"; a11y check passes.
- **Spec reference:** US-07; NFR — "Privacy / GDPR."

### T5.5 — Cookies page and consent banner

- **Description:** Build (a) the `/cookies` informational page and (b)
  a minimal cookie consent banner that defaults to "no tracking" until
  the visitor accepts. The banner is only required if the chosen
  analytics vendor (Open Question 5; default Plausible) needs consent;
  if Plausible is used and configured cookieless, the banner can be
  omitted but the cookies page still ships.
- **Acceptance criteria:**
  - [ ] `/cookies` page renders Danish copy explaining what cookies (if
        any) the site sets.
  - [ ] If consent is needed, the banner renders on first visit only
        (state stored in `localStorage`).
  - [ ] Banner contains exactly two actions: "Accepter" and "Afvis".
        No third "Manage" button in v1.
  - [ ] Banner does not block interaction with the rest of the page
        (does not cover the CTA).
  - [ ] Until "Accepter" is clicked, no analytics events fire.
  - [ ] If the analytics vendor is genuinely cookieless and consent is
        not legally required, the banner is omitted; this decision is
        recorded in the PR description with a link to the legal basis.
- **Definition of done:** End-to-end test asserts no analytics events
  fire before consent (when banner is in use); page renders; banner
  is keyboard-operable.
- **Spec reference:** Out of Scope — "Cookie-consent flows beyond the
  legal minimum required for analytics"; NFR — "Privacy / GDPR."

## Epic 6 — Price Widget

**Goal:** Build the reusable live-rate widget that calls the backend
price API, displays the current øre/kWh including markup with a
timestamp, and degrades gracefully when the backend is unreachable or
the data is stale. CC BY 4.0 attribution to "Energi Data Service /
Energinet" is shown wherever the widget appears.

**Dependencies:**
- Epic 1 complete.
- Backend price API endpoint must be available in staging. Per ADR-001
  §3.4, the data is served from local storage on the backend (daily
  cron at 13:15 Europe/Copenhagen ingests Energi Data Service); the
  marketing site never calls Energi Data Service or energinet.dk
  directly.

**Tasks:**

### T6.1 — Backend price API client

- **Description:** Build a small client module that fetches the
  current rate from the backend price API. Module is server-side
  (called during SSR or from a route handler) to keep API base URLs
  configurable per environment and to enable response caching.
- **Acceptance criteria:**
  - [ ] Client reads the backend base URL from an environment
        variable.
  - [ ] Client sends a GET to the documented price endpoint with a
        timeout of 5 seconds.
  - [ ] Response is parsed into a typed shape:
        `{ priceOreKwh: number, updatedAt: ISO8601 string, area: "DK1"|"DK2" }`
        (or whatever the backend contract specifies — confirm with
        backend team).
  - [ ] Errors (timeout, non-2xx, malformed JSON) return a single
        typed `PriceUnavailable` value rather than throwing.
  - [ ] Module is unit-tested against a mocked fetch with at least
        these cases: success, 500, timeout, malformed JSON.
- **Definition of done:** Backend team has reviewed the consumed
  endpoint contract; unit tests pass; module is used by T6.2.
- **Spec reference:** Technical Context — "Live price data source:
  Internal pricing service"; ADR-001 §3.4.

### T6.2 — Live rate widget UI component

- **Description:** Build the widget that renders the current rate, the
  "Opdateret kl. HH:MM" timestamp, and the explainer "Prisen følger
  spotprisen på det danske elmarked". Refreshes every 15 minutes while
  the page is open or on page refresh.
- **Acceptance criteria:**
  - [ ] On first render, the widget displays the rate fetched
        server-side (so it appears in initial HTML — supports SEO and
        LCP).
  - [ ] On client mount, the widget schedules a refresh every 15
        minutes via `setInterval` (cleared on unmount).
  - [ ] Rate is rendered formatted to da-DK conventions:
        "X,XX kr./kWh" (or "XX øre/kWh" — final unit chosen by
        product, single content constant).
  - [ ] Timestamp is rendered as "Opdateret kl. HH:MM" using `da-DK`
        time formatting.
  - [ ] Explainer text "Prisen følger spotprisen på det danske
        elmarked" appears beside the rate.
  - [ ] Widget renders correctly at 360px width without truncation.
- **Definition of done:** Component is used in the hero (Epic 2) and
  optionally in the pricing section; visual regression snapshots
  cover all three states (fresh, stale, unavailable) per T6.3 and T6.4.
- **Spec reference:** US-03 — "Visitor sees the current live kWh rate."

### T6.3 — Stale data state

- **Description:** When the backend reports the rate was updated more
  than 1 hour ago, render the rate with a muted timestamp and a small
  warning icon, but do not block the page.
- **Acceptance criteria:**
  - [ ] Widget detects staleness based on `updatedAt` from the API,
        not the time of the HTTP call.
  - [ ] When stale, the timestamp uses a muted token colour and a
        warning icon (inline SVG, with `aria-label="Prisen er måske
        ikke helt aktuel"`).
  - [ ] The rate itself is still displayed.
  - [ ] No alert, no modal — just the muted styling.
  - [ ] Unit test covers boundary: 59 minutes old (fresh), 61 minutes
        old (stale).
- **Definition of done:** End-to-end test mocks an API response with
  a 2-hour-old timestamp and asserts the warning state renders.
- **Spec reference:** Error States — "Live price data is stale (last
  update > 1 hour ago)."

### T6.4 — Unavailable state with verbatim copy

- **Description:** When the backend price API errors or times out,
  render the widget's fallback message instead of the rate. Per spec,
  show "Aktuel pris hentes…" for up to 5 seconds, then "Prisen er
  midlertidigt utilgængelig — prøv igen om lidt".
- **Acceptance criteria:**
  - [ ] On initial fetch failure, widget shows "Aktuel pris hentes…"
        for 5 seconds.
  - [ ] After 5 seconds with no successful response, widget switches
        to "Prisen er midlertidigt utilgængelig — prøv igen om lidt".
  - [ ] Both copy strings are exact (verbatim from spec) and live in
        the content module.
  - [ ] The rest of the page (hero, CTA, pricing) is unaffected.
  - [ ] When JavaScript is disabled, widget shows "Aktiver JavaScript
        for at se den aktuelle pris" via a `<noscript>` block.
  - [ ] CTA button continues to function in all of the above states.
- **Definition of done:** End-to-end tests cover (a) timeout, (b)
  500 response, (c) `<noscript>` rendering; copy strings asserted
  verbatim.
- **Spec reference:** Error States — "Live price data unavailable" and
  "JavaScript disabled in browser."

### T6.5 — CC BY 4.0 attribution

- **Description:** Display attribution to "Energi Data Service /
  Energinet" wherever the live rate is shown, per ADR-001 §3.4.
- **Acceptance criteria:**
  - [ ] Below or beside the widget, attribution text reads:
        "Datakilde: Energi Data Service / Energinet (CC BY 4.0)" with
        a link to the Energi Data Service website.
  - [ ] Attribution renders in a smaller token font size; meets
        minimum body contrast.
  - [ ] Attribution link opens in a new tab with
        `rel="noopener noreferrer"`.
  - [ ] Attribution appears in every placement of the widget.
- **Definition of done:** End-to-end test asserts attribution text and
  link presence wherever the widget is rendered.
- **Spec reference:** ADR-001 §3.4 — "CC BY 4.0 attribution to
  'Energi Data Service / Energinet' is displayed wherever the data
  appears."

**Goal:** Hit Core Web Vitals targets in production, expose

  - [ ] Hero image (if any) is served as AVIF/WebP with a fallback,

## Epic 8 — Error States

**Goal:** Implement every error scenario in the spec's Error States
table with the verbatim Danish copy and the prescribed behaviour, and
ship a 404 page.

**Dependencies:**
- Epic 6 (price unavailable behaviour is already covered in T6.4).
- Epic 4 (signup unreachable fallback already covered in T4.3).
- Epic 1, 2 complete (so a 404 page can use the layout).

**Tasks:**

### T8.1 — 404 page

- **Description:** Build the custom 404 page in Danish with friendly
  copy and links back to home and to signup.
- **Acceptance criteria:**
  - [ ] Page renders for any unmatched route under the marketing
        domain.
  - [ ] Page returns HTTP 404 (not 200) per SEO best practice.
  - [ ] Copy is in Danish (final copy by product; placeholder:
        "Siden findes ikke. Måske er linket forkert, eller siden er
        flyttet.").
  - [ ] Two links: "Til forsiden" → `/`; "Tilmeld dig" → onboarding URL.
  - [ ] Page uses the same layout (header, footer) as all other pages.
  - [ ] Page is server-rendered.
- **Definition of done:** Curl against `/this-does-not-exist` returns
  status 404 with the page body; visual regression snapshot stored.
- **Spec reference:** Error States — "404 (unknown URL): Custom 404
  page in Danish with link back to home and to signup."

### T8.2 — Unsupported-browser banner

- **Description:** Detect very old browsers (no `IntersectionObserver`
  support, no ES2020) and render a non-blocking banner suggesting an
  upgrade. The site must still render.
- **Acceptance criteria:**
  - [ ] On an unsupported browser, a banner appears at the top of the
        page with Danish copy: "Din browser er gammel. For den bedste
        oplevelse, opdater til en nyere version."
  - [ ] Banner has a dismiss button.
  - [ ] Banner does not block any content (does not cover the hero or
        the CTA).
  - [ ] On supported browsers, no banner appears.
  - [ ] Detection runs without breaking the page on the very old
        browsers themselves (defensive feature detection, not
        `try/catch` around modern syntax).
- **Definition of done:** Manual test against an old browser (e.g.
  IE11 emulation or an old WebView) confirms the banner renders and
  the rest of the page still works.
- **Spec reference:** Error States — "User on an unsupported old
  browser: A non-blocking banner suggests upgrading; site still
  renders."

### T8.3 — Disabled-CTA state when signup service is known to be down

- **Description:** When operations need to disable signup (e.g.
  scheduled maintenance), allow them to flip a single env variable
  that disables every CTA across the site and replaces it with a
  message.
- **Acceptance criteria:**
  - [ ] An env variable (`NEXT_PUBLIC_SIGNUP_DISABLED=true` or
        equivalent) toggles the state.
  - [ ] When disabled, all `<SignupCTA />` instances render as a
        non-clickable element with the Danish copy: "Tilmelding er
        midlertidigt nede — prøv igen om lidt."
  - [ ] When disabled, the sticky mobile CTA is hidden (no point
        showing it).
  - [ ] When the variable is unset or `false`, the CTA functions
        normally.
  - [ ] Toggling the variable does not require a code change — only a
        redeploy via host configuration.
- **Definition of done:** Manual test on staging: flip the env var,
  trigger redeploy, confirm CTAs render the disabled state; flip back,
  confirm restored. End-to-end test runs against both states.
- **Spec reference:** Error States — "Signup service / Onboarding Flow
  URL unreachable" (extended interpretation: planned outage path).

### T8.4 — Error states regression test suite

- **Description:** Bundle all the verbatim-copy error states from the
  spec into a single end-to-end test file so a future copy change
  fails the build loudly.
- **Acceptance criteria:**
  - [ ] Test file exercises: price unavailable (T6.4), price stale
        (T6.3), signup unreachable fallback page (T4.3), 404 (T8.1),
        signup-disabled state (T8.3), `<noscript>` price widget
        (T6.4).
  - [ ] Each test asserts the verbatim Danish string from the spec.
  - [ ] Test file is referenced from the spec in a small "verified by"
        section in this epics document or in `docs/specifications/`.
  - [ ] Tests run in CI on every PR.
- **Definition of done:** All tests pass on `main`; deliberately
  introducing a typo in the copy is shown to fail the build.
- **Spec reference:** Error States table (entire) — copy fidelity is
  testable.
