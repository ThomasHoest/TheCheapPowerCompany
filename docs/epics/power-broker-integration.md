## Overview

The Power Broker Integration component is the backend boundary between The Cheap Power Company and the Danish electricity market. It delivers (a) a daily Energi Data Service ingestion job that mirrors DK1 and DK2 day-ahead Elspot prices into local storage, (b) a date-versioned tariff and tax configuration plus a calculation engine that produces the all-in øre/kWh consumer price, (c) Eloverblik third-party API integration for customer-consented meter data access, (d) the regulatory artefact tracking required to operate as a registered elleverandør in DataHub, and (e) an internal REST endpoint that serves current, forecast, and historical prices to the iOS app, marketing site, and admin portal. The critical path runs E-01 to E-02 to E-05 in parallel with E-03 (Eloverblik) and E-04 (DataHub registration ops, which gates production launch but not development). BRS-H1 supplier-switch ebIX integration is explicitly out of scope here per the spec; ADR-001 section 6.2 flags this as a separate critical deliverable to be owned elsewhere.

## E-01 — Energi Data Service Ingestion

Deliver a reliable scheduled job that fetches DK1 and DK2 day-ahead Elspot prices once daily, persists them to local storage with retries and alerts on failure, and exposes ingestion health to the admin portal — by end of epic, local storage holds 24 DK1 and 24 DK2 hourly records for D+1 by 13:30 CET on every normal day. Depends on backend service runtime and local storage (per Backend/API specification, ADR-001 section 6.3), available logging/alerting infrastructure, and a time zone library configured for Europe/Copenhagen.

- [ ] **T-0101** Implement the `spot_price` table/collection per spec section 6.1 with fields `hourUTC`, `hourDK`, `priceArea`, `spotPriceDKK`, `spotPriceEUR`, `ingestedAt`, `source`, a unique natural key on (`hourUTC`, `priceArea`), and `priceArea` constrained to `DK1` or `DK2`. Provide a repository upsert function keyed on the natural key that returns inserted/updated/unchanged counts; second insert with the same key updates rather than duplicates; invalid `priceArea` rejected with a domain error; negative `spotPriceDKK` accepted as-is. Migration committed and applied to test environment, repository unit-tested, code review approved. Satisfies US-01, US-02, US-04 (immutability enforced at T-0105).
- [ ] **T-0102** Build a thin HTTP client wrapper for `https://api.energidataservice.dk/dataset/Elspotprices` per spec section 7.1 with query parameters `filter={"PriceArea":["DK1","DK2"]}`, `start=StartOfDay`, `end=StartOfDay+P2D`, `limit=96`, `timezone=dk`, `sort=HourDK desc`. Issue a single GET, parse the JSON into typed records (`HourUTC`, `HourDK`, `PriceArea`, `SpotPriceDKK`, `SpotPriceEUR`); HTTP non-2xx raises typed `EdsHttpError` (with status, URL, body); 200 with empty `records` array raises `EdsEmptyResultError` (per spec section 9); network/timeout failures raise `EdsTransportError`; no retry logic in this client (fail-fast — retries live in T-0104); request timeout configurable, default 30 seconds. Unit tests cover happy path, 4xx, 5xx, empty array, and timeout via stubbed transport; deployed to test environment. Satisfies US-01.
- [ ] **T-0103** Compose T-0101 and T-0102 into an `IngestSpotPricesJob` that fetches today and tomorrow for DK1+DK2 and upserts results in a single transaction (or equivalent atomic operation); after a successful run, DB contains 24 DK1 and 24 DK2 hourly records for D+1; emits a structured INFO log with `timestamp`, `requestUrl`, `httpStatus`, `recordsPersisted`, `durationMs`, `runId`; records run outcome (success/failure + reason) in `ingestion_run_log` for the health dashboard (used in T-0505); existing records not duplicated and not overwritten by default (immutability in T-0105). Integration test against stubbed Energi Data Service exercises full run end to end; deployed to test environment; manual run verified to populate D+1. Satisfies US-01.
- [ ] **T-0104** Schedule `IngestSpotPricesJob` daily at 13:15 Europe/Copenhagen (verified in both winter CET and summer CEST settings); on transient failure (5xx, transport, empty array) retry at +5, +15, +30, +60 minutes from last attempt; after all retries within a 4-hour window are exhausted, raise an on-call alert with run identifier and most recent error; 4xx non-empty response raises an on-call alert within 15 minutes and does not retry; successful retry cancels any pending alert and logs recovery; concurrency-safe — a second scheduled trigger while the first is in flight does not start a parallel run. Retry/alert logic unit-tested with a virtual clock; integration test simulates 5xx-then-200 sequence; alert delivery verified to on-call channel in test environment. Satisfies US-01, US-08.
- [ ] **T-0105** Per US-04 acceptance criterion 4, enforce historical immutability: modify upsert to leave existing rows untouched by default for any `hourUTC` whose `hourDK` date is in the past relative to Europe/Copenhagen "today"; normal scheduled run for D and D+1 does not modify any row whose `hourDK` < D; attempted upsert against an existing past row produces an `unchanged` result rather than `updated`; add a `--force` invocation flag (used only by manual administrative re-ingestion) that permits overwrite and logs `forceOverwrite=true`; default scheduled invocation always runs `force=false`; empty-result responses never overwrite existing data (per spec section 9). Unit tests cover both default and force modes for past-date rows; deployed to test environment. Satisfies US-04.
- [ ] **T-0106** Implement a once-weekly backfill job (per spec section 7.1) running at a configurable time (default Sunday 03:00 Europe/Copenhagen) that scans the last 90 days for missing (`hourUTC`, `priceArea`) entries (expecting 24 per area per day); identifies any day where DK1 or DK2 has fewer than 24 hourly rows; issues a targeted GET to Energi Data Service per such day and upserts the missing rows; honours immutability (existing rows not modified); if source still does not return data for a known gap, logs the gap and raises an alert after two consecutive weekly failures for the same gap; logs run summary with days scanned, gaps detected, gaps filled, gaps still open. Unit and integration tests cover no-gap, gap-filled, gap-still-open cases; deployed to test environment with manual trigger entry point. Satisfies US-01, US-04 (gap detection), US-08 indirectly.
- [ ] **T-0107** Per spec section 10 and ADR-001 section 3.4, ensure every API response that exposes Elspot data carries CC BY 4.0 attribution metadata: add a top-level `attribution` object to the price API response with `source: "Energi Data Service / Energinet"`, `licence: "CC BY 4.0"`, `licenceUrl: "https://creativecommons.org/licenses/by/4.0/"`; centralise the attribution string in one constant (no consumer hard-codes it); document the attribution requirement in the API contract and that downstream UI must display it; log the attribution in the ingestion run record for traceability. Contract test asserts presence on every endpoint variant; API docs updated; reviewed by Architect. Satisfies US-01 supporting requirement; spec section 10 (NFR: Attribution).

## E-02 — Consumer Price Calculation Engine

Provide a date-versioned tariff and tax configuration store plus a deterministic calculation engine that, given an `hourUTC`, `priceArea`, and optionally `dsoAreaCode`, returns the all-in consumer price in øre/kWh broken down into components, including the 1 January 2026 elafgift transition and time-of-use network tariffs. Depends on E-01 T-0101 (spot price storage) and Backend/API specification configuration access-control decisions (ADR-001 section 6.3).

- [ ] **T-0201** Implement the versioned configuration table per spec section 6.2 with fields `componentType`, optional `priceArea`, optional `dsoAreaCode`, `validFrom` (inclusive), optional `validTo` (exclusive; null = open-ended), `value` in øre/kWh excluding VAT (precision ≥ 4 decimal places), and optional `hourOfDayPattern`; `componentType` constrained to enum `supplier_margin`, `network_tariff`, `energinet_system`, `energinet_balancing`, `elafgift`; constraint or validation rule prevents two overlapping rows for the same (`componentType`, `priceArea`, `dsoAreaCode`) where `[validFrom, validTo)` intervals overlap; expose an auth-gated admin-API write path for inserts/updates and a read path for the calculation engine. Migration applied; CRUD repository unit-tested including overlap rejection; deployed to test environment. Satisfies US-07.
- [ ] **T-0202** Implement `getActiveComponent(componentType, priceArea, dsoAreaCode, hourUTC)` returning the configured value valid at the given hour, or null if no configuration covers it: returns the row whose `[validFrom, validTo)` interval contains queried `hourUTC` (converted to Europe/Copenhagen for date comparison); when no row matches, returns null and the caller decides handling (per T-0204); when `hourOfDayPattern` is present, lookup interprets the pattern to pick the correct band; pattern syntax documented in `/docs/architecture/tariff-pattern-syntax.md` (or equivalent existing location) and supports at least peak/off-peak split by hour-of-day-DK and weekend vs weekday distinction. Unit tests cover boundary cases at `validFrom`, at `validTo - 1 hour`, daylight-saving transitions, and time-of-use band changes. Satisfies US-07.
- [ ] **T-0203** Implement the calculation engine per spec section 6.4 — inputs: `hourUTC`, `priceArea`, optional `dsoAreaCode`; output: structured result containing `spotOreKwh`, `supplierMargin`, `energyPrice` (= spot + margin), `networkTariff`, `energinetSystem`, `energinetBalancing`, `elafgift`, `subtotalExVat`, `vat`, `finalConsumerPrice`; formula `final = (energy + network + system + balancing + elafgift) × 1.25`; spot conversion `øre/kWh = SpotPriceDKK / 10`; supplier margin added algebraically (negative spot prices preserved, not floored); all intermediate values exposed for UI breakdown; VAT (25%) applied to the sum, never to each component individually; numerical precision ≥ 4 decimal places throughout (rounding for display is the consumer's responsibility); when supplier margin is missing from configuration, return error code `SUPPLIER_MARGIN_NOT_CONFIGURED` (operational invariant: there must always be exactly one active supplier margin); function is pure (no I/O other than configuration and spot lookup). Unit tests verify the full formula against at least three hand-computed examples covering positive spot, near-zero spot, and negative spot; deployed to test environment. Satisfies US-03, US-07.
- [ ] **T-0204** Per spec section 9 and US-07, enforce missing-network-tariff error path: calling all-in calculation for `(hourUTC, priceArea, dsoAreaCode)` whose network tariff is unconfigured returns error code `NETWORK_TARIFF_NOT_CONFIGURED` with a payload identifying the missing area code and queried hour, suitable for direct rendering by the iOS app ("Cannot compute total price for your area — please contact support"); a separate `calculateEnergyPrice(hourUTC, priceArea)` returns spot + margin without DSO configuration and without VAT, used by the price API when DSO is unknown (e.g. marketing site); error code documented in API contract. Unit tests cover both all-in failure case and energy-only success case. Satisfies US-07, spec section 9 row "Network tariff for the consumer's DSO area is missing".
- [ ] **T-0205** Per spec section 6.2 and Open Question 5, configure the elafgift date-versioned transition for 1 January 2026: two `elafgift` configuration rows — one with `validFrom` < 2026-01-01 and `validTo = 2026-01-01`, another with `validFrom = 2026-01-01` and `validTo` null (reduced rate ~1 øre/kWh, exact figure to be confirmed by Finance); a unit test computes the all-in price for an hour at 2025-12-31 23:00 DK and 2026-01-01 00:00 DK and asserts the elafgift component differs; re-running calculation for any pre-2026 hour after configuration is in place yields the same value as before the transition row was added; admin-portal-visible report lists the active elafgift value for any given date. Configuration rows present in test environment with placeholder values; unit tests pass; figure confirmation tracked as Open Question 5 in the source spec. Satisfies US-07 (date-versioning criterion).
- [ ] **T-0206** Provide an authenticated admin-only API for the active `supplier_margin` configuration: `GET /admin/pricing/supplier-margin` returns currently active row and next scheduled row (if any); `POST /admin/pricing/supplier-margin` accepts `validFrom` (must be in the future or today) and `value`, inserts a new row that closes the previous open-ended row by setting its `validTo` to the new `validFrom`; gated by an admin role (per ADR-001 section 3.8); attempt to insert with `validFrom` in the past rejected with `MARGIN_HISTORICAL_WRITE_FORBIDDEN`; all writes recorded in audit log with actor, before/after values, timestamp. Endpoint deployed to test environment; admin auth enforced; audit log entries verified; OpenAPI/contract docs updated. Satisfies US-03 (markup configuration), spec section 10 (Auditability).

## E-03 — Eloverblik Third-Party Integration

Enable customers to grant The Cheap Power Company access to their consumption data via Eloverblik's third-party API; by epic end, an authenticated customer can complete the consent flow, the platform fetches their hourly consumption and charges, and consent withdrawal is detected and handled gracefully. Depends on Eloverblik third-party registration (regulatory/ops prerequisite tracked separately), customer authentication via Signicat OIDC (ADR-001 section 3.2), and the secrets store for the 1-year refresh token (ADR-001 section 3.5: client devices never see Eloverblik tokens).

- [ ] **T-0301** Provide encrypted-at-rest storage for the 1-year Eloverblik refresh token (in the secrets store referenced by ADR-001 section 3.6) and admin tooling to rotate it: token never readable by any client device; admin CLI/endpoint installs a new refresh token, old token overwritten only after the new one verified to mint an access token; automated check warns on-call engineer at least 30 days before refresh-token expiry; token retrieval access-logged with calling subsystem; unit/integration tests verify no public endpoint exposes the refresh token. Token storage deployed to test environment with sandbox refresh token; rotation procedure documented in `/docs/runbooks/eloverblik-token-rotation.md`; security review approved. Satisfies US-05, spec section 10 (Privacy).
- [ ] **T-0302** Implement refresh-token-to-access-token exchange against Eloverblik with 24-hour access token cached in memory (or short-lived secret store), reused until within 1 hour of expiry then refreshed proactively; if access token rejected with 401 (per spec section 9), attempt exactly one re-exchange from the refresh token; if that also fails, refresh token treated as expired (handled in T-0307); access tokens never logged in clear text; concurrent callers do not trigger multiple parallel exchanges (single-flight or equivalent); access tokens never persisted beyond their natural lifetime. Unit tests cover happy path, single 401 retry, double 401 escalation, and concurrent caller behaviour; tested against Eloverblik sandbox using a sandbox refresh token; deployed to test environment. Satisfies US-05.
- [ ] **T-0303** Implement the consent flow per spec section 7.3: `POST /customers/{id}/eloverblik/consent-start` returns a redirect URL to Eloverblik scoped to the third-party app and the customer's selected metering point/window; after the customer returns, `POST /customers/{id}/eloverblik/consent-confirm` calls `GET /api/meteringpoints/meteringpoints` and persists `meteringPointId`, `address`, `gridArea`, `meterType`, `consentValidFrom`, `consentValidTo`, `consentStatus = granted` (per spec section 6.5); if the polled list does not include the newly-consented point within a configurable window (default 5 minutes), confirmation endpoint returns `CONSENT_NOT_VISIBLE_YET` and iOS may retry; all Eloverblik calls use the data access token from T-0302 as a Bearer token in the `Authorization` header; consent flow runs in `ASWebAuthenticationSession` on iOS (ADR-001 section 3.5) — backend never embeds Eloverblik credentials in the redirect; audit log entries written for consent-start and consent-confirm with customer ID, metering point ID, and result. End-to-end test against Eloverblik sandbox covers the round-trip; deployed to test environment; iOS and backend contract documented. Satisfies US-05.
- [ ] **T-0304** Implement a service function and scheduled job that calls `POST /api/meterdata/gettimeseries/{dateFrom}/{dateTo}/Hour` for granted metering points, parses hourly consumption (kWh), and persists records per spec section 6.6: function accepts `meteringPointId`, `dateFrom`, `dateTo` and returns parsed records; records upserted into `eloverblik_consumption` keyed on (`meteringPointId`, `hourUTC`) — existing values not silently overwritten unless re-fetch explicitly requested; scheduled job runs daily per customer to re-fetch the previous 7 days (per spec section 7.2 default TTL); on 5xx, retry with exponential backoff up to 1 hour (per spec section 9); on 403 for a metering point, set `consentStatus = withdrawn` and stop scheduled fetches (handled in T-0306); on any failure, customer-facing app surfaces "consumption data may be delayed" notice via a flag in the price API response (touch point with E-05). Sandbox-tested for happy path, 5xx retry, and 403 handling; deployed to test environment. Satisfies US-05.
- [ ] **T-0305** Implement a service function that calls `POST /api/meterdata/getcharges` for granted metering points and parses the charge breakdown (tariff, subscription, fee) for use by the calculation engine (E-02) when configured network tariffs are unavailable for a DSO area: function returns parsed charges with effective windows and unit values; charges persisted with provenance (`source = eloverblik:getcharges`) and `fetchedAt` timestamp; queryable by metering point and effective date; if `getcharges` fails with 5xx, function returns most recent cached payload along with `stale: true` indicator and `lastSuccess` timestamp; ingestion runs at most once per metering point per day under normal operation. Sandbox-tested; persisted records observable in test environment; unit-tested. Satisfies US-05.
- [ ] **T-0306** Handle consent revocation: a 403 response from any metering-point-scoped Eloverblik call sets `consentStatus = withdrawn` for that metering point; no further scheduled fetches issued for a withdrawn metering point until consent is re-granted (re-running the consent flow from T-0303 reactivates fetches); a notification is generated for the customer via the standard notification channel and the iOS app surfaces this on next launch; state transition recorded in audit log with timestamp and triggering API response. Unit and integration tests cover withdrawal-then-regrant lifecycle; lifecycle test passes against the sandbox; deployed to test environment. Satisfies US-05, spec section 9 ("Eloverblik returns 403").
- [ ] **T-0307** Per spec section 9, when refresh token is detected as expired (direct error from token exchange or after two consecutive 401s on access-token exchange — see T-0302): expiry detected within one failed exchange cycle; all metering points whose customers depend on the refresh token marked `needs_reconsent` (state machine: `granted` → `needs_reconsent`); raise on-call alert with severity HIGH; customers receive a notification prompting re-consent; after a new refresh token is installed via T-0301 tooling, scheduled fetches resume on next cycle without code changes. Failure-injection test verifies the lifecycle; deployed to test environment with a runbook entry in `/docs/runbooks/eloverblik-token-rotation.md`. Satisfies US-05, spec section 9 ("Eloverblik refresh token expired").

## E-05 — Price API Endpoint

Provide an internal REST API that serves current, forecast, and historical øre/kWh prices to the iOS app, marketing site, and admin portal — all reads served from local storage (per ADR-001 section 3.4); the API never calls Energi Data Service synchronously; includes appropriate cache headers, the `stale: true` flag when data older than 26 hours, historical-gap reporting from US-04, and attribution metadata from T-0107. Depends on E-01 (spot price storage and ingestion), E-02 (calculation engine), and Backend/API specification auth model for internal endpoints (ADR-001 section 6.3).

- [ ] **T-0501** Implement `GET /prices/current?priceArea={DK1|DK2}&dsoAreaCode={optional}` returning the price for the current hour and remaining hours of today plus all of tomorrow if available — response includes per-hour `spotOreKwh`, `energyPrice` (spot + margin), and `finalConsumerPrice` when DSO is supplied (energy-only otherwise per T-0204); `priceArea` required (missing or invalid returns HTTP 400 with code `PRICE_AREA_REQUIRED` or `PRICE_AREA_INVALID` per US-02); when `dsoAreaCode` omitted the response contains energy-only prices (no `finalConsumerPrice`), when supplied it contains the full breakdown per T-0203; response includes the attribution object from T-0107; each hour returned with at least 4 decimal places of precision (per US-03); rows include both raw spot and spot+margin (per US-03 acceptance criterion 3); negative spot prices returned as-is (per US-03); response p95 latency under 200 ms for a warmed cache against local storage (per spec section 10). Endpoint deployed to test environment; contract test passes; latency budget verified under representative load. Satisfies US-02, US-03, US-07, US-08 (stale flag added in T-0503).
- [ ] **T-0502** Implement `GET /prices/history?priceArea={DK1|DK2}&from={iso}&to={iso}&page={n}` returning hourly records from local storage — required parameters `priceArea`, `from`, `to` (missing or invalid returns HTTP 400); when range exceeds 31 days the response is paginated with default page size 1000 records and `nextPageToken` when more data exists; when any hour in the requested range is missing from local storage the response includes a `gaps` array listing each missing `hourUTC` (rather than silently omitting); records returned ordered ascending by `hourUTC` within a page; response includes the attribution object; an `Authorization`-gated variant (or separate parameter) controls whether the all-in calculation is included — for unauthenticated marketing-site reads only the spot+margin energy price is returned. Endpoint deployed to test environment; pagination, gap reporting, and ordering covered by integration tests. Satisfies US-04.
- [ ] **T-0503** Per US-08 and spec section 9, every price API response must include a `stale: true` flag whenever the most recent persisted record for the requested area is older than 26 hours — when stale, the response also includes `lastIngestedAt` so the iOS app can render "Live prices temporarily unavailable — last updated [timestamp]"; all price endpoints (T-0501, T-0502, T-0505) compute and return `stale` and `lastIngestedAt`; `stale = true` if and only if `now - lastIngestedAt > 26 hours` for the requested `priceArea`; API never silently substitutes a default, average, or extrapolated value for a missing hour (per US-08 acceptance criterion 3); when stale, no HTTP error is returned — caller still receives whatever data is available with the flag. Integration tests cover three cases (fresh, exactly-at-threshold, stale); flag verified across all endpoints; iOS and marketing-site contract teams notified; documentation updated. Satisfies US-08.
- [ ] **T-0504** Set HTTP cache headers appropriate per endpoint so clients (iOS, marketing site) and any intermediate CDN can honour them without showing stale data past the SLO: `GET /prices/current` returns `Cache-Control: public, max-age=300, stale-while-revalidate=600` (or equivalent values agreed with Architect) and an `ETag`; `GET /prices/history` returns `Cache-Control: public, max-age=3600, immutable` for ranges entirely in the past and `max-age=300` for ranges that include today; responses honour `If-None-Match` and return 304 when `ETag` matches; when `stale: true` is set, `Cache-Control` is reduced to `max-age=60` so a clearing of the upstream issue propagates quickly; `Vary: Accept-Encoding` (and any locale header used) is set correctly. Headers verified by integration tests; CDN configuration documented; deployed to test environment. Satisfies US-08, spec section 10 (Latency).
- [ ] **T-0505** Per US-08, expose ingestion freshness via `GET /admin/health/spot-ingestion` returning, for each of DK1 and DK2: `lastIngestedAt`, `lastSuccessfulRunId`, `lastFailureAt` (if any), `lastFailureReason` (if any), `currentlyStale` (boolean) — endpoint gated by an admin role; data sourced from the `ingestion_run_log` populated in T-0103; admin-portal page renders the values with a green/amber/red indicator (green = fresh, amber = > 22 hours, red = > 26 hours / stale); manual trigger for an out-of-band ingestion run (per T-0103) is exposed and gated by the admin role. Endpoint and dashboard deployed to test environment; reviewed by operations. Satisfies US-08.
